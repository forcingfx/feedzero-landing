Fixed

First-launch error path no longer silently completes.

If the browser couldn't initialize the encrypted database (Web Crypto unavailable), onboarding was being marked complete anyway. The user landed in a half-initialized state instead of seeing the error.

Affected

Users on browsers without Web Crypto.

The most common path was iOS Safari with Lockdown Mode enabled. Modern browsers were unaffected, which is why this slipped past notice for several releases.

iOS Lockdown Mode no SubtleCrypto
Privacy-extreme browsers disabled
Safari, Chrome, Firefox always worked
The fix

One guard before completing onboarding.

Asynchronous failures in the init store set error instead of throwing. The handler now checks for that error before promoting the user to a returning-user state.

// src/app.tsx .then(() => { if (!useAppStore.getState().error) completeOnboarding(); })
Caught by

The test that had been failing all along.

An app-init test for the error path was already in the suite. It had been failing across multiple releases without being investigated. This release makes it green and gives the path real coverage.

Principle

Errors must fail loud, not mark progress.

If a step in an init chain fails, downstream steps should not run. The previous code treated “promise resolved” as “step succeeded,” even when the resolved value carried an error. Result types are only useful if every caller checks them.

✓Result types over thrown errors
✓Every chain checks for prior failure
✓Tests guard the unhappy path
What stays solid

The rest of the app, unchanged.

This release is one targeted fix. No new features, no behavior changes for the 99.9% of users on browsers with Web Crypto. The mobile drawer, stats page, sync, and everything from 0.7.0 carry forward.
